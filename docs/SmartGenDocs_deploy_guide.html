SmartGen Docs: Deployment Guide

This guide provides step-by-step instructions to deploy your own instance of SmartGen Docs—the autonomous, AI-powered documentation generation platform. By the end of this guide, you will have a fully functioning GitOps pipeline that scans target repositories using Gemini 3.5 Flash, generates responsive documentation, and hosts it automatically via GitHub Pages.

Prerequisites

Before starting, ensure you have the following:

  1. GitHub Account: To host the repository, execute GitHub Actions, and serve the UI via GitHub Pages.
  2. Google AI Studio API Key: Required to access the Gemini 3.5 Flash model.
  3. Obtain an API key from Google AI Studio.
  4. GitHub Personal Access Token (PAT) (Optional/Recommended): If you plan to run local tests or interact with private repositories, a PAT with repo scope is useful. Otherwise, the default GitHub Actions token will suffice.

Step-by-Step Deployment

1. Fork or Clone the Repository

Create your own copy of the repository to establish your GitOps workspace:

# Clone the repository
git clone https://github.com/bayzed123/SmartGenDocs.git
cd SmartGenDocs

2. Configure Local Environment (Optional: For Local Testing)

To test the generation engine locally before deploying it to GitHub Actions:

  1. Install Python Dependencies: Ensure you have Python 3.8+ installed, then run: bash pip install -r requirements.txt

  2. Set Environment Variables: Configure your Gemini API key:

  3. Linux/macOS: bash export GEMINI_API_KEY="your-gemini-api-key-here"

  4. Windows (PowerShell): powershell $env:GEMINI_API_KEY="your-gemini-api-key-here"

  5. Run the Script: bash python generate_docs.py

3. Configure GitHub Repository Secrets

To run the generation process automatically inside GitHub Actions, you must store your API keys securely.

  1. On your GitHub repository page, navigate to Settings > Secrets and variables > Actions.
  2. Click New repository secret.
  3. Add the following secret:
  4. Name: GEMINI_API_KEY
  5. Value: Your Google AI Studio API key

4. Enable Read and Write Permissions for Workflows

The automation pipeline needs permission to commit the newly generated HTML files back to your repository.

  1. Navigate to your repository's Settings > Actions > General.
  2. Scroll down to Workflow permissions.
  3. Select Read and write permissions.
  4. Check Allow GitHub Actions to create and approve pull requests (if prompted/desired).
  5. Click Save.

5. Configure GitHub Pages

The web portal and the generated documentation will be hosted for free via GitHub Pages.

  1. Navigate to Settings > Pages.
  2. Under Build and deployment:
  3. Source: Select Deploy from a branch.
  4. Branch: Select main (or your default branch) and the folder / (root).
  5. Click Save.
  6. Once configured, your documentation portal will be live at: https://<your-username>.github.io/SmartGenDocs/

Verifying the Deployment

Let's test the automated GitOps workflow to ensure everything is connected properly.

  1. Navigate to the input_links/ directory in your repo.
  2. Create a new text file named test-repo.txt.
  3. Paste a public GitHub repository link inside it (e.g., https://github.com/octocat/Spoon-Knife).
  4. Commit and push the changes to your main branch.
  5. Click on the Actions tab in your repository. You should see a workflow run start automatically.
  6. Once the workflow successfully completes:
  7. Check the docs/ folder to ensure the HTML files (Overview, Architecture, Deployment Guide, How to Use) have been generated.
  8. Visit https://<your-username>.github.io/SmartGenDocs/ to see your live documentation site updated with the newly scanned repository.

Troubleshooting

Issue Cause Resolution
Workflow fails on write/commit step Incorrect workflow permissions. Ensure Workflow permissions in Actions settings is set to Read and write permissions.
API Authentication Error (401/403) Missing or invalid GEMINI_API_KEY. Verify that the secret is correctly named GEMINI_API_KEY and contains the correct API token from Google AI Studio.
No documentation output Target repository is private or link is malformed. Ensure target URLs are public and follow the https://github.com/owner/repository structure.
GitHub Pages returns 404 Site not yet built or wrong branch selected. Ensure GitHub Pages source is configured to the main branch and the root directory. Allow 1-2 minutes for the initial build to complete.
