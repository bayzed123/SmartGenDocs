Deployment Guide: Auto Sitemap Generator

This deployment guide provides comprehensive, step-by-step instructions for deploying and automating the Auto Sitemap Generator (sitemap-generator-saas v2.0.0).

Because this application contains a static frontend (frontend/) and a Node.js-based crawling backend (backend/) powered by Puppeteer, deployment is divided into hosting the frontend interface and running/scheduling the crawler backend.

1. Prerequisites

Before deploying to any platform, ensure your environment meets the minimum requirements:

  • Node.js: v14.0.0 or higher (v18+ recommended)
  • npm: v6.x or higher
  • Operating System: Linux (Ubuntu/Debian recommended for headless Chrome), macOS, or Windows.
  • Puppeteer System Dependencies: Puppeteer requires a browser engine (Chromium) and several underlying OS-level libraries to run in headless mode.

2. Deployment Option A: Automated GitHub Actions & GitHub Pages (Recommended)

This hybrid approach allows you to host the user-facing dashboard on GitHub Pages for free, while executing the backend crawler automatically using GitHub Actions on a schedule (e.g., daily or weekly).

Step 1: Deploy the Frontend to GitHub Pages

  1. Go to your cloned repository on GitHub.
  2. Navigate to Settings > Pages.
  3. Under Build and deployment, select:
  4. Source: Deploy from a branch
  5. Branch: main (or your default branch)
  6. Folder: /frontend (If you want to serve only the frontend folder. Alternatively, keep root / and configure path redirection).
  7. Click Save. Your frontend will be live at https://<your-username>.github.io/SitemapGenerator/.

Step 2: Configure the GitHub Actions Workflow (crawler.yml)

Create a workflow file at .github/workflows/crawler.yml to run the crawler, generate the sitemap, and commit the outputs back to your repository.

name: Run Auto Sitemap Crawler

on:
  schedule:
    # Runs at 00:00 every Sunday (Modify as needed)
    - cron: '0 0 * * 0'
  workflow_dispatch: # Allows manual trigger via GitHub Actions tab

jobs:
  crawl:
    runs-on: ubuntu-latest

    steps:
    - name: Checkout repository
      uses: actions/checkout@v3

    - name: Set up Node.js
      uses: actions/setup-node@v3
      with:
        node-level: '18'
        cache: 'npm'

    - name: Install dependencies
      run: npm install

    - name: Run Crawler
      run: npm start

    - name: Commit and Push generated Sitemap
      run: |
        git config --local user.email "github-actions[bot]@users.noreply.github.com"
        git config --local user.name "github-actions[bot]"
        git add output/sitemap.xml output/links.json output/progress.json
        git commit -m "chore: auto-update sitemap [skip ci]" || echo "No changes to commit"
        git push
      env:
        GITHUB_TOKEN: ${{ secrets.GITHUB_TOKEN }}

Step 5: Automate with PM2 or Cron

To set up a cron job that runs the crawler every night at midnight: 1. Open the crontab manager: bash crontab -e 2. Add the following line (adjust path to your project directory): text 0 0 * * * cd /path/to/SitemapGenerator && /usr/bin/npm start >> /path/to/SitemapGenerator/output/cron.log 2>&1

4. Deployment Option C: Platform-as-a-Service (Render / Railway)

If you wish to wrap this crawler backend in an API server (e.g., using Express) and host it in the cloud, you can deploy it to Render or Railway.

Deploying to Render

  1. Create a new Web Service on Render and link your repository.
  2. Set the following Build and Start settings:
  3. Environment: Node
  4. Build Command: npm install
  5. Start Command: npm start (Make sure your code implements an HTTP listener if running as a persistent Web Service).
  6. Puppeteer Configuration: Render supports Puppeteer out of the box, but you must add the Render Puppeteer Buildpack or use a custom Dockerfile to ensure Chromium system dependencies are present.
  7. In your Render dashboard, navigate to Settings > Buildpacks.
  8. Add the following buildpack URL: https://github.com/jontewks/puppeteer-heroku-buildpack.git

5. Puppeteer Troubleshooting & Performance Tuning

When running the crawler in production environments, standard headless browser limitations apply.

1. "Failed to launch the browser process!" Error

If you encounter launching issues on Linux, modify the Puppeteer launch arguments in backend/crawler.js to run in no-sandbox mode:

// Inside backend/crawler.js (browser initialization)
const browser = await puppeteer.launch({
  headless: 'new',
  args: [
    '--no-sandbox',
    '--disable-setuid-sandbox',
    '--disable-dev-shm-usage',
    '--disable-gpu'
  ]
});

2. High Memory Usage / Crashing

  • The Problem: Puppeteer spawns a Chromium process that consumes significant memory during long crawls.
  • The Fix: Limit concurrency or implement paging limits inside your Breadth-First Search (BFS) logic. Utilize progress.json to keep track of state, allowing the crawler to save its progress, exit periodically to free up system memory, and resume safely.
