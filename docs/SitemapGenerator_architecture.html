System Architecture Guide: Auto Sitemap Generator

This document provides a comprehensive overview of the design patterns, system topology, component interactions, and data flows of the Auto Sitemap Generator. This system is structured as an automated crawler-and-generator designed to map web properties, respect compliance requirements, and compile standard-compliant XML sitemaps.

1. Architectural Blueprint & System Topography

The application is structured as a decoupled architecture containing a Dynamic Crawler Worker (Backend Engine), a Stateless Presentation Interface (Frontend Dashboard), and a Structured Persistence Layer (File-system Output).

2. Decoupled Subsystem Breakdown

2.1 Backend Core (Crawler & Parser Engine)

The backend is a Node.js process run via a task executor or GitHub Actions workflow. Rather than a static HTTP request crawler, this system leverages a headless browser to ensure compatibility with client-side rendered (CSR) applications (React, Vue, Angular, etc.).

  • crawler.js (The Orchestrator): Manages the central Breadth-First Search (BFS) lifecycle. It initializes the browser instance, manages concurrency limits, coordinates queue iteration, and hands crawled content off to parsers.
  • sitemap.js (Sitemap Generation Utility): Formats mapped links into standard-compliant XML using standard schemas (<urlset>, <url>, <loc>, etc.).
  • progress.js (Checkpoint Manager): Monitors system memory, handles state tracking, and writes checkpoints to prevent loss of crawler progression in the event of an unhandled exception or crash.

2.2 Client Interface (Frontend Dashboard)

A decoupled, lightweight user interface hosted on GitHub Pages. It enables configuration validation and serves as a telemetry client.

  • index.html & style.css: Structure and visual themes for managing crawler configurations and displaying live queue telemetry.
  • app.js & dashboard.js: Consume local progress JSON payloads to render status updates, crawl velocities, queue states, and visual site topology trees.

2.3 Persistence Layer (File System Output)

To avoid the architectural overhead of a running relational database, execution data is maintained on-disk: * links.json: Raw flat array of discovered interior URLs. * progress.json: Serialized dump of the current traversal queue and visited status hashes. * sitemap.xml: The finalized SEO asset distributed to production servers.

4. Key Architectural Dependencies & Technical Decisions

4.1 Rendering Engine: Headless Chrome via Puppeteer

  • Decision: The runtime engine executes raw Chromium rendering loops ("puppeteer": "^22.0.0").
  • Rationale: Standard Node.js crawlers utilizing raw HTML fetches (like Axios/Cheerio) fail on modern Single Page Applications (SPAs). Using Puppeteer ensures client-side rendered routes are fully generated in the virtual DOM before link extraction begins.

4.2 Traversal Strategy: In-Memory Graph vs Memory Safety

To prevent out-of-memory (OOM) exceptions on large domains, the application balances memory usage with disk writes: * The Queue and Visited hashes reside in-memory for immediate lookup lookup ($O(1)$ complexity). * During active runs, the engine periodically flushes state to disk (progress.json) through progress.js, providing an off-heap recovery plan in case of crash.

5.2 Structured Schema Generation (sitemap.xml)

The XML generator standardizes output formatting:

<?xml version="1.0" encoding="UTF-8"?>
<urlset xmlns="http://www.sitemaps.org/schemas/sitemap/0.9">
  <url>
    <loc>https://example.com/</loc>
    <lastmod>2023-10-27</lastmod>
    <changefreq>daily</changefreq>
    <priority>1.0</priority>
  </url>
</urlset>

6. Execution Guidelines

Production Workload Profiling

For scalable production workflows, configure Puppeteer flags within backend/crawler.js to minimize system footprints:

// Recommended configuration for headless optimization
const browser = await puppeteer.launch({
  headless: 'new',
  args: [
    '--no-sandbox',
    '--disable-setuid-sandbox',
    '--disable-dev-shm-usage', // Prevents memory constraints in containerized runtimes
    '--disable-gpu'
  ]
});

To limit resource usage, block asset pipelines (e.g., stylesheets, images, fonts) during navigation events to focus purely on anchor extraction.
